# CLAUDE.md

このリポジトリで作業する際のガイドです。

## プロジェクト概要

頭の中で考えていることやタスクがごちゃごちゃした時に、それを「看板（カード）」として全部書き出し、いる・いらないを仕分けることで頭の中を整理するセルフケアツール「頭整理くん（atama-seiri-kun）」。
`index.html` をブラウザで開くだけで動作する。サーバー・ビルド・パッケージ管理・依存ライブラリは一切なし。

- 主要ファイル: [`index.html`](index.html) — HTML/CSS/JS をすべて内包した単一ファイル
- 仕様書: [`spec.md`](spec.md) — 目的・体験フロー・機能・データ仕様の定義
- データ永続化: ブラウザの `localStorage`（キー: `atama-seiri-kun:cards`）。外部送信は行わない。
- 共有・バックアップ: 当面なし（データは端末内のみ。ユーザーが任意で全消去できる）。

目的（詳細は [`spec.md`](spec.md) 参照）:

- ぐるぐる思考や「次、何やればいいんだっけ？」となったときに、頭の中を全部外に出して整理する
- いらないものを捨て、**ポジティブなもの・本当にやるべきことだけが残った頭の状態**に変える

設計思想（重要）: **「整理術」ではなく「自分と向き合うための儀式」**。判断を強制せず、本人の手で1枚ずつ仕分けることに意味を置く。残ったものを見て「これだけでいいんだ」と安心できる体験を最優先する。新たなプレッシャー（やらなきゃ感）を生まない。

## 構成

すべて `index.html` 一枚に収める方針。

- `<style>` — 全UIのCSS（CSS変数でテーマ管理）。落ち着いた・前向きなトーン、余白多め、レスポンシブ対応。
- `<script>` — カードの状態を保持し、localStorage へ保存する
  - カード配列キー: `atama-seiri-kun:cards`
  - 1エントリ: `{id, text, status, createdAt}`
    - `status` … カードの状態（`inbox`=出したまま未仕分け / `keep`=いる / `trash`=いらない＝ゴミ箱）
    - `text` … カードの内容
    - `createdAt` … 追加時刻

### 体験フロー（3ステップ）

```
[STEP 1] 全部だす → [STEP 2] 仕分ける → [STEP 3] 残す
```

1. **STEP 1: 全部だす（ブレインダンプ）** — 頭の中にあるものをカードとしてとにかく羅列する。良し悪しや重複を気にせず、入力ハードル最小（Enter連打で追加できる等）。
2. **STEP 2: 仕分ける（トリアージ）** — カードを1枚ずつ見て「いる（keep）」「いらない（trash＝ゴミ箱）」に振り分ける。判断できないものは保留（inbox のまま）でよい。ゴミ箱から戻せる。
3. **STEP 3: 残す（クリアな頭）** — 「いる」カードだけが残った状態を表示。これが「整理後の頭の中」。達成感・安心感を演出する。

- 全リセット: すべてのカードをクリアして最初からやり直せる（確認あり）。

### 設計上の配慮

- **安心感を最優先**: 残ったものを見て気持ちが軽くなる体験にする。煽り・競争・他者比較・件数表示はしない。
- **プレッシャーを与えない**: 仕分けを強制しない。「やらなければ」と思わせる文言を避ける。
- **捨てる行為のハードルを下げる**: ゴミ箱に捨てても戻せる安心感を担保する。
- **プライバシー**: データは端末内のみ。いつでも全消去できることを明示。
- **堅牢性**: localStorage が使えない環境でも、保存なしで動作継続（クラッシュさせない。読み込み失敗時は空配列にフォールバック）。
- **アクセシビリティ**: 十分なコントラスト、キーボード操作可、system font stack。

## 動作確認

ビルド不要。`index.html` をブラウザで直接開くか、任意の静的サーバーで配信する。

```
npx --yes http-server -p 8777 -c-1
```

（`python` はこの環境では Windows ストアのスタブで動かないため Node を使う。）

## 開発ワークフロー（必須ルール）

以下は**必ず守る**こと:

1. **作業は機能単位で Issue を立ててから始める。** 実装の前に GitHub Issue を作成し、何を作るか・受け入れ条件を明記する。
2. **作業はブランチを切って行い、PR を出す。** 直接 `main` にコミットしない。PR 本文に `Closes #<issue番号>` を入れて Issue と紐付ける。
3. **PR を出した後、必ず `/code-review` を実施する。** 指摘事項はマージ前に対応するか、対応しない場合は PR コメント or 別 Issue として残す。
4. **`/code-review` の指摘は次のサイクルで処理する。**
   1. 指摘内容を PR コメントとして残す（対応する／しないの方針も記載）。
   2. 対応すると判断した指摘を修正してコミット・プッシュする。
   3. 再度 `/code-review` を実施する。
   4. 新たな指摘がなくなるまで 1〜3 を繰り返す。問題がなければマージ確認に移る。

### 補足

- コミット・プッシュ・PR作成・マージは、ユーザーが明示的に指示したときのみ行う。
- コミットメッセージ・PR・Issue は日本語で記述する。
- 単一ファイル構成を崩さない（外部依存・ビルド工程を持ち込まない）ことを基本方針とする。
